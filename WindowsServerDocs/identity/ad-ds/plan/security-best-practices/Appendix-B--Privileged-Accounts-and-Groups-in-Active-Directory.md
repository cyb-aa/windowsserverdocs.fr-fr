# <a name="appendix-b-privileged-accounts-and-groups-in-active-directory"></a>Annexe B : Comptes privilégiés et groupes dans Active Directory

>S'applique à : Windows Server 2016, Windows Server 2012 R2, Windows Server 2012


## <a name="appendix-b-privileged-accounts-and-groups-in-active-directory"></a>Annexe B : Comptes privilégiés et groupes dans Active Directory  
« Privilégiés » des comptes et groupes dans Active Directory sont ceux à laquelle des autorisations, des privilèges et droits puissantes sont accordées qui lui permettre d’effectuer presque n’importe quelle action dans Active Directory et sur les systèmes joints à un domaine. Cette annexe commence par une description des droits, des privilèges, et les autorisations, suivi d’informations sur le « nombre le plus élevé du privilège » comptes et groupes dans Active Directory, autrement dit, les plus puissants et les groupes.  

Fournit également des informations sur les comptes et groupes dans Active Directory, en plus de leurs droits intégrés et par défaut. Bien que les recommandations de configuration spécifique pour sécuriser les comptes à privilèges plus élevés et les groupes sont fournies en tant qu’annexes distincts, cette annexe fournit des informations générales qui vous permet d’identifier les utilisateurs et groupes, que vous devez vous concentrer sur la sécurisation . Pour ce faire, vous devez, car ils peuvent être exploitées par des attaquants de compromettre et même détruire votre installation Active Directory.  

### <a name="rights-privileges-and-permissions-in-active-directory"></a>Droits, privilèges et autorisations dans Active Directory  
Les différences entre les droits, les autorisations et privilèges peuvent être confuse et contradictoire, même au sein de la documentation de Microsoft. Cette section décrit certaines des caractéristiques de chacun d’eux lorsqu’ils sont utilisés dans ce document. Ces descriptions pas prévaudra pour d’autres documents de Microsoft, car il peut utiliser ces termes différemment.  

#### <a name="rights-and-privileges"></a>Droits et privilèges  
Droits et privilèges sont efficacement les mêmes fonctionnalités de l’échelle du système qui sont accordées aux principaux de sécurité tels que les utilisateurs, les services, les ordinateurs ou les groupes. Dans les interfaces généralement utilisées par les professionnels de l’informatique, ceux-ci sont généralement appelés « rights » ou « droits utilisateur », et ils sont souvent affectés par les objets de stratégie de groupe. La capture d’écran suivante montre certaines des droits utilisateur courants qui peuvent être attribuées aux principaux de sécurité (elle représente la stratégie par défaut des contrôleurs de domaine dans un domaine Windows Server 2012). Certaines de ces droits s’appliquent à Active Directory, telles que la **permettre aux comptes d’ordinateur et utilisateur à être approuvé pour délégation** droit utilisateur, tandis que les autres droits s’applique au système d’exploitation Windows, telles que **modifier le système temps**.

#### <a name="permissions"></a>Autorisations  
Les autorisations sont des contrôles d’accès qui sont appliquées aux objets sécurisables tels que le système de fichiers, Registre, service et les objets Active Directory. Chaque objet sécurisable possède une liste de contrôle d’accès associées (ACL), qui contient les entrées de contrôle d’accès (ACE) qui accordent ou refusent des principaux de sécurité (utilisateurs, services, ordinateurs ou groupes) de la capacité à effectuer diverses opérations sur l’objet. Par exemple, les listes ACL pour de nombreux objets dans Active Directory contiennent des entrées ACE qui autorise les utilisateurs authentifiés lire des informations générales sur les objets, mais ne leur accordez pas la possibilité de lire les informations sensibles ou pour modifier les objets.
À l’exception compte invité de chaque domaine, chaque principal de sécurité qui se connecte et est authentifié par un contrôleur de domaine dans une forêt Active Directory ou d’une forêt approuvée a l’authentifié aux utilisateurs identificateur de sécurité (SID) ajouté à son accès jeton par défaut. Par conséquent, si un utilisateur, un service ou un compte d’ordinateur tente de lire les propriétés générales sur les objets utilisateur dans un domaine, l’opération de lecture a réussi.  

Si un tentatives de principal de sécurité pour accéder à un objet sans les entrées sont définies et qui contient un SID présent dans le jeton d’accès du principal, le principal ne peut pas accéder à l’objet. En outre, si une ACE dans l’ACL d’un objet contient une entrée de refus pour un SID qui correspond au jeton d’accès de l’utilisateur, l’ACE de « refus » sera généralement remplacer un conflit lié à « autoriser » ACE. Pour plus d’informations sur le contrôle d’accès dans Windows, consultez [contrôle d’accès](https://msdn.microsoft.com/library/aa374860(v=VS.85).aspx) sur le site Web MSDN.  

Dans ce document, les autorisations fait référence aux fonctionnalités qui sont accordées ou refusées aux principaux de sécurité sur les objets sécurisables. Chaque fois qu’il existe un conflit entre un droit d’utilisateur et une autorisation, le droit d’utilisateur est généralement prioritaire. Par exemple, si un objet dans Active Directory a été configuré avec une ACL qui refuse les administrateurs de tous les accès en lecture et écriture à un objet, un utilisateur qui est membre du groupe d’administrateurs du domaine ne pourra pas afficher autant d’informations sur l’objet. Toutefois, étant donné que le groupe Administrateurs est accordé à l’utilisateur droite « prendre possession de fichiers ou d’autres objets », l’utilisateur peut simplement prendre possession de l’objet en question, puis réécrire les ACL de l’objet pour accorder aux administrateurs un contrôle complet de l’objet.  

Il est pour cette raison que ce document vous encourage à éviter à l’aide de comptes puissants et des groupes pour l’administration quotidienne, plutôt que d’essayer de limiter les fonctionnalités des comptes et des groupes. Il n’est pas efficacement possible d’empêcher un utilisateur déterminé, qui a accès aux informations d’identification puissantes d’utiliser ces informations d’identification pour accéder à n’importe quelle ressource sécurisable.  

### <a name="built-in-privileged-accounts-and-groups"></a>Intégré privilégié des comptes et groupes  
Active Directory est destiné à faciliter la délégation de l’administration et le principe du moindre privilège lors de l’attribution des droits et autorisations. « Regular » aux utilisateurs qui disposent de comptes dans un domaine Active Directory sont, par défaut, en mesure de lire une grande partie de ce qui est stocké dans le répertoire, mais sont en mesure de modifier uniquement un ensemble très limité de données dans le répertoire. Les utilisateurs qui nécessitent des privilèges supplémentaires peuvent être accordées à l’appartenance à différents groupes privilégiés qui sont intégrées à l’annuaire afin qu’ils peuvent effectuer des tâches spécifiques liées à leurs rôles, mais ne peut pas effectuer des tâches qui ne sont pas pertinentes pour leurs tâches.  

Dans Active Directory, il existe trois groupes intégrés qui composent les groupes de privilège le plus élevés dans le répertoire : le groupe Administrateurs de l’entreprise (EA), le groupe Admins du domaine (DA) et le groupe intégré Administrateurs (BA).  

Une quatrième groupe, le groupe Administrateurs du schéma (SA), dispose de privilèges qui, si l’objet d’abus, peuvent endommager ou de détruire un ensemble de la forêt Active Directory, mais ce groupe n’est plus dans ses fonctionnalités que les clients EA, DA et BA groupes restreint.  

Outre ces quatre groupes, il existe un nombre d’intégrées supplémentaires et des comptes par défaut et des groupes dans Active Directory, chacun d’eux est accordé des droits et autorisations qui permettent à effectuer des tâches administratives spécifiques. Bien que cette annexe ne fournit pas une étude approfondie de chaque groupe intégré ou par défaut dans Active Directory, il fournit une table des groupes et des comptes que vous allez probablement voir dans vos installations.  

Par exemple, si vous installez Microsoft Exchange Server dans une forêt Active Directory, groupes et comptes supplémentaires peuvent créés dans les conteneurs intégrés et les utilisateurs dans vos domaines. Cette annexe décrit uniquement les groupes et les comptes qui sont créés dans les conteneurs intégrés et les utilisateurs dans Active Directory, basé sur les fonctionnalités et rôles natifs. Comptes et groupes qui sont créés par l’installation de logiciels d’entreprise ne sont pas inclus.  

#### <a name="enterprise-admins"></a>Administrateurs de l’entreprise  
Le groupe Administrateurs de l’entreprise (EA) se trouve dans le domaine racine de forêt, et par défaut, il s’agit d’un membre du groupe Administrateurs intégré dans chaque domaine dans la forêt. Le compte administrateur intégré dans le domaine racine de forêt est le seul membre par défaut du groupe EA. EAs bénéficient de droits et autorisations qui leur permettent d’affecter des modifications à la forêt. Il s’agit des modifications qui affectent tous les domaines de la forêt, comme l’ajout ou suppression de domaines, l’établissement d’approbations de forêt ou augmentation des niveaux fonctionnels de forêt. Dans un modèle de délégation correctement conçu et implémenté, l’appartenance au contrat entreprise est obligatoire uniquement lors de la construction de tout d’abord la forêt ou lorsque vous apportez certaines modifications de la forêt telles que l’établissement d’une approbation de forêt sortante.  

Le groupe EA se trouve par défaut dans le conteneur utilisateurs dans le domaine racine de forêt, et c’est un groupe de sécurité universel, sauf si le domaine racine de forêt est en cours d’exécution en mode mixte Windows 2000 Server, auquel cas le groupe est un groupe de sécurité global. Bien que certains droits sont accordées directement au groupe de contrat entreprise, de nombreuses les droits de ce groupe sont réellement héritées par le groupe EA, car il est membre du groupe Administrateurs dans chaque domaine dans la forêt. Administrateurs de l’entreprise n’ont aucuns droits par défaut sur les stations de travail ou des serveurs membres.  

#### <a name="domain-admins"></a>Administrateurs du domaine  
Chaque domaine dans une forêt possède son propre groupe Admins du domaine (DA), qui est un membre du groupe d’administrateurs (BA) intégré de ce domaine en plus d’un membre du groupe Administrateurs local sur chaque ordinateur est joint au domaine. Le seul membre par défaut du groupe DA pour un domaine est le compte administrateur intégré pour ce domaine.  

DAs sont tout puissants dans leurs domaines, tandis que EAs ont des privilèges de la forêt. Dans un modèle de délégation correctement conçu et implémenté, DA doit être appartenir uniquement dans les scénarios de « secours », qui sont des cas dans lequel un compte avec des privilèges élevés sur chaque ordinateur dans le domaine est nécessaire, ou lorsque certains l’ensemble du domaine modifications doivent être effectuées. Bien que les mécanismes de délégation Active Directory natifs autorisent la délégation dans la mesure où il est possible d’utiliser des comptes de DA uniquement dans les scénarios d’urgence, la construction d’un modèle de délégation effective peut prendre du temps et de nombreuses organisations utilisent des fournisseurs tiers applications pour accélérer le processus.  

Le groupe de DA est un groupe de sécurité global situé dans le conteneur utilisateurs du domaine. Il existe un groupe de DA pour chaque domaine dans la forêt et le seul membre par défaut d’un groupe de DA est le compte d’administrateur intégré du domaine. Étant donné que le groupe de DA d’un domaine est imbriqué dans le groupe du domaine BA et groupe d’administrateurs local de chaque système joints au domaine, DAs ont non seulement les autorisations qui sont spécifiquement accordées aux administrateurs de domaine, mais ils héritent également tous les droits et les autorisations accordées à la groupe d’administrateurs du domaine et le groupe Administrateurs local sur tous les systèmes joints au domaine.  

#### <a name="administrators"></a>Administrateurs  
Le groupe intégré Administrateurs (BA) est un groupe de domaine local dans le conteneur d’intégré d’un domaine dans lequel DAs et EAs sont imbriqués, et il est à ce groupe qui bénéficie de la plupart des droits d’accès directs et des autorisations dans le répertoire et sur les contrôleurs de domaine. Toutefois, le groupe Administrateurs pour un domaine ne dispose pas des privilèges sur les serveurs membres ou sur les stations de travail. L’appartenance au groupe Administrateurs local d’ordinateurs joints au domaine est où local est accordée ; et des groupes abordés, DAs uniquement sont membres du groupe Administrateurs locaux de tous les ordinateurs joints au domaine par défaut.  

Le groupe Administrateurs est un groupe de domaine local dans le conteneur intégré du domaine. Par défaut, chaque BA groupe contient le compte d’administrateur intégré du domaine local, groupe de DA du domaine local et EA groupe du domaine racine de forêt. Nombre de droits d’utilisateur dans Active Directory et sur les contrôleurs de domaine sont accordées spécifiquement au groupe Administrateurs, pas à EAs ou DAs. Groupe de BA d’un domaine est accordé des autorisations Contrôle total sur la plupart des objets d’annuaire et peut prendre possession d’objets d’annuaire. Bien que les groupes EA et DA bénéficient de certaines autorisations spécifiques à un objet dans la forêt et les domaines, la puissance des groupes est réellement « héritée » à partir de leur appartenance à des groupes de BA.  

> [!NOTE]  
> Bien que ce sont les configurations par défaut de ces groupes privilégiés, un membre de l’un des trois groupes peut manipuler le répertoire pour devenir membre d’aucun des autres groupes. Dans certains cas, il est facile à réaliser, tandis que dans d’autres cas, il est plus difficile, mais du point de vue du privilège potentiel, tous les trois groupes doivent être considérées comme équivalentes efficacement.  

#### <a name="schema-admins"></a>Administrateurs du schéma  
Le groupe Administrateurs du schéma (SA) est un groupe universel dans le domaine racine de forêt et a un compte d’administrateur intégré de ce domaine uniquement comme membre par défaut, comme le groupe EA. Bien que l’appartenance au groupe administrateur système peut permettre à un attaquant de compromettre le schéma Active Directory, qui est l’infrastructure pour la forêt Active Directory, SAP ont peu de droits par défaut et d’autorisations au-delà du schéma.  

Vous devez soigneusement gérer et surveiller l’appartenance au groupe administrateur système, mais à certains égards, ce groupe est « moins privilégié » que les trois plus élevés privilégié groupes décrits précédemment, car l’étendue de son privilège est très étroite ; Autrement dit, les SAP n’ont aucuns droits d’administration n’importe où autre que le schéma.
